# 标准发布规范

> **版本**: v2.0  
> **适用范围**: 所有软件项目。核心闸口项目无关。  
> **哲学**: 发布是产品面对世界的时刻——没有"差不多"，没有"跳过这条"。  
> **最后更新**: 2026-06-22

---

## 1. 发布哲学

发布不是"代码写完了顺手打个 tag"。发布是一道不可逆的闸口——一旦推送，全世界的用户、审计者、安全研究者都能看到你的代码和产物。

**四条铁律**：

1. **没有"跳过"**。检查清单的每一条都必须执行。没有"这次来不及，下个版本补"。
2. **敏感信息零容忍**。任何形式的密钥、token、内部路径、个人信息，都不得进入 git 历史。
3. **四人全部签字**。产品负责人、架构师、PM、工程师——少一个人签字，不得发布。
4. **发布后不可撤销**。tag 可以被 force-push，但 Git 历史和他人 clone 不会被改写。发布了就是发布了。

---

## 2. 角色与发布职责

| 角色 | 发布职责 | 签字域 |
|------|---------|--------|
| **产品负责人 (Jobs)** | 愿景对齐、用户体验完整、端到端可用 | 最终发布签字 |
| **软件架构师** | 架构合规、安全审计、敏感信息扫描 | 安全闸口签字 |
| **产品经理** | 功能完整、验收通过、Release Notes 准确 | 功能签字 |
| **资深研发工程师** | 编译通过、测试全部通过、二进制可运行 | 技术签字 |

**禁止事项**：
- 工程师不得在自己的模块未通过 Review 时签字
- 架构师不得在敏感信息扫描通过前签字
- PM 不得在验收清单有未通过项时签字
- Jobs 不得在其他三人签字完成前签字

## 3. 敏感信息管控

### 3.1 禁止进入 Git 的文件（强制）

| 类别 | 文件 | .gitignore 规则 |
|------|------|----------------|
| 密钥 | `.env` | `.env` |
| 内部文档 | `发布规范.md`, `开发日志.md`, `验收规范.md`, `开发规范.md` | 文件名精确匹配 |
| 编译产物 | `goalos-daemon`, `goalos`, `goalos-cli` | `/goalos-daemon` 等根目录匹配 |
| 发布脚本 | `RELEASE.md`, `upload-release.sh` | 精确匹配 |
| 二进制 | `plugin-shell`, `plugin-websearch` | `/plugin-shell` |

### 3.2 禁止进入 Git 的内容（强制）

| 模式 | 示例 | grep 检测 |
|------|------|----------|
| API Key | `sk-ant-...`, `ghp_...` | `grep -rE "ghp_\|sk-\|x-api-key\|Bearer [A-Za-z0-9]"` |
| 密码 | `password=`, `passwd=` | `grep -rE "password\|passwd\|secret="` |
| 内部路径 | `/Users/haochen/` | `grep -r "/Users/"` |
| Token 明文 | `151e8e...` | `grep -rE "[0-9a-f]{20,}"` |

### 3.3 发布前敏感信息扫描（强制，不可跳过）

```bash
# 必须全部返回空
grep -rE "ghp_|sk-ant-|x-api-key|Bearer [A-Za-z0-9]{20,}" . --include="*.go" --include="*.md" --include="*.sh" --include="*.yaml" --include="*.yml" | grep -v ".git/" | grep -v ".env"
grep -r "/Users/" . --include="*.go" --include="*.sh" --include="*.md" | grep -v ".git/"
```

**任一条有输出 → 发布阻塞 ❌。不得继续。**

## 4. 版本号规范

采用语义化版本：`MAJOR.MINOR.PATCH`

| 类型 | 触发条件 | 示例 |
|------|---------|------|
| **MAJOR** | 架构变更、不兼容 API 修改 | v1→v2: Executor 独立 OS 子进程 |
| **MINOR** | 新功能、里程碑完成、向后兼容 | v1.0→v1.1: websearch Plugin |
| **PATCH** | Bug 修复、安全补丁 | v1.1.0→v1.1.1: Token 格式兼容修复 |

- 版本号严格递增。不跳号。不重用。
- 每次发布必须更新开发日志中的版本号和发布日期。

## 5. 发布检查清单（强制，逐条执行）

### Phase 1: 技术闸口（Engineer + Architect）

| # | 检查项 | 命令 | 通过标准 |
|---|--------|------|---------|
| 1 | 编译通过 | `go build ./...` | 零错误 |
| 2 | 全量测试通过 | `go test -count=1 -timeout 120s -race ./...` | 零失败、零 race |
| 3 | 静态分析 | `go vet ./...` | 零 error |
| 4 | 跨平台编译 | Linux/Darwin/Windows × amd64/arm64 | 全部通过 |
| 5 | 敏感信息扫描 | §3.3 两条命令 | 全部返回空 |
| 6 | Stub 审计 | `grep -rn "不验证" internal/ cmd/` | 零结果 |
| 7 | 内部文档未追踪 | `git ls-files \| grep -E "发布规范\|开发日志\|验收规范\|开发规范"` | 零结果 |

**Phase 1 全部通过后 → Engineer + Architect 签字 → 进入 Phase 2**

### Phase 2: 功能闸口（PM + Architect）

| # | 检查项 | 方法 | 通过标准 |
|---|--------|------|---------|
| 8 | 验收清单通过 | 对照验收规范 §6 行为验收清单 | 全部 ✅ |
| 9 | 接线检查通过 | 对照验收规范 §8 | 4/4 通过 |
| 10 | 点火测试通过 | 编译 daemon → 启动 → curl → 检查日志 | 日志含 GoalCompleted |
| 11 | Release Notes 准确 | 对照 git log 检查 | 每条变更有对应说明 |

**Phase 2 全部通过后 → PM + Architect 签字 → 进入 Phase 3**

### Phase 3: 发布闸口（Jobs + All）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 12 | 四人全部签字 | Engineer ✓ + Architect ✓ + PM ✓ + Jobs ✓ |
| 13 | Tag 已创建 | `git tag -l 'v*'` 含新版本 |
| 14 | CI 已触发 | GitHub Actions workflow run 状态 = success |
| 15 | Release 已创建 | GitHub Release 页面可见，含 5 平台附件 |

**Phase 3 全部通过后 → Jobs 最终签字 → ✅ 发布完成**

## 6. Git 提交规范

### 6.1 提交格式

```
<type>: <简短描述>

- 详细说明
- 设计依据（引用架构文档或决议编号）

Co-Authored-By: Claude <noreply@anthropic.com>
```

### 6.2 Type 类型

| Type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档 |
| `chore` | 构建/工具 |
| `refactor` | 重构 |
| `test` | 测试 |
| `security` | 安全修复 |

### 6.3 提交粒度

- 一个提交 = 一个逻辑变更
- 提交前 `go build ./... && go test -race ./...` 必须通过
- 禁止提交包含敏感信息的文件

## 7. Tag 管理

### 7.1 创建 Tag

```bash
git tag -a v1.0.3 -m "GoalOS v1.0.3 — {简短描述}"
git push origin v1.0.3
```

### 7.2 Tag 命名

- 格式：`v<MAJOR>.<MINOR>.<PATCH>`
- 禁止：`v1`、`release-1`、`1.0.3`

### 7.3 Tag 保护

- Tag 创建后禁止 force-push（GitHub Settings → Tags → Protected）
- 若必须修复（go.sum 缺失等）→ force-push 需要架构师 + Jobs 双重批准
- Tag 触发 CI 自动执行测试 → 构建 → Release 创建 → Docker 推送

## 8. Release 管理

### 8.1 Release 内容

每个 Release 必须包含：

| 内容 | 说明 |
|------|------|
| **Release Notes** | 新功能、修复、已知问题 |
| **多平台二进制** | darwin-amd64、darwin-arm64、linux-amd64、linux-arm64、windows-amd64 |
| **Docker 镜像引用** | `ghcr.io/{owner}/{repo}:{version}` |

### 8.2 Release 自动化

推送 `v*` tag → GitHub Actions 自动：
1. `go vet` + `go test -race`
2. 5 平台二进制构建 + 打包
3. GitHub Release 创建（含附件）
4. Docker 镜像构建 + 推送至 ghcr.io

---

## 9. 紧急修复流程

当生产环境发现严重 bug 需要立即修复时：

1. 创建 `hotfix/{描述}` 分支
2. 修复 + 测试（不可跳过任何 Phase 1 检查）
3. PM 确认修复范围（只修这个 bug，不混入其他变更）
4. 合并 → PATCH 版本升级 → 完整发布流程

**紧急修复不是跳过检查的理由。Phase 1-3 全部照常执行。**
