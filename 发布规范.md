> **版本**: v1.0  
> **最后更新**: 2026-06-20  
> **适用范围**: 所有 GoalOS 发布操作

---

## 1. 版本号规范

采用语义化版本（Semantic Versioning）：`MAJOR.MINOR.PATCH`

| 类型 | 说明 | 示例 |
|------|------|------|
| **MAJOR** | 架构变更。不兼容的 API 修改 | v1→v2: Executor 改为独立 OS 子进程 |
| **MINOR** | 新功能。向后兼容 | v1.0→v1.1: 新增 Persona 系统 |
| **PATCH** | Bug 修复。向后兼容 | v1.1.0→v1.1.1: 修复 EventBus panic |

**规则**：
- 每个里程碑完成 → MINOR 版本升级
- 架构变更（05 架构文档 MAJOR 升级） → MAJOR 版本升级
- 紧急 Bug 修复 → PATCH 版本升级
- 版本号严格递增。不跳号。不重用

## 2. Git 提交规范

### 2.1 提交格式

```
<type>: <简短描述>

- 详细说明（可选。多行）
- 设计依据（可选。引用架构文档或决议编号）
```

### 2.2 Type 类型

| Type | 用途 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: Event Bus + State Store — 核心基础设施` |
| `fix` | Bug 修复 | `fix: Snapshot daemon集成修复` |
| `docs` | 文档 | `docs: 用户手册 + 开发规范` |
| `chore` | 构建/工具 | `chore: 初始化 Go 模块与 gitignore` |
| `refactor` | 重构 | `refactor: Scheduler 状态机拆分` |
| `test` | 测试 | `test: 新增 Governance 集成测试` |
| `security` | 安全修复 | `security: Token 验证移回 daemon 侧` |

### 2.3 提交粒度

- 一个提交 = 一个逻辑变更
- 一个模块的完整实现 = 一个提交（含测试）
- 不混合无关变更
- 提交前 `go build ./... && go test ./...` 必须通过

### 2.4 提交历史

- 12 次提交的提交结构（当前 v1.0.1）：
  1. `chore` — 项目初始化
  2. `feat` — 事件类型 + 错误码
  3. `feat` — EventBus + StateStore
  4. `feat` — Scheduler + GoalAnchor
  5. `feat` — Governance + MissionEngine
  6. `feat` — PluginRunner + IPC
  7. `feat` — Daemon API + Client + Config
  8. `feat` — Persona + Channel + ContextEngine
  9. `feat` — Daemon + CLI + E2E 测试
  10. `docs` — 用户手册 + 开发规范 + 验收规范 + 开发日志
  11. `fix` — daemon main.go 模块注册
  12. `fix` — Snapshot 集成 + 开发日志更新

## 3. Tag 管理

### 3.1 创建 Tag

```bash
# 创建带注释的 tag
git tag -a v1.0.1 -m "GoalOS v1.0.1 — 多平台 + Docker (ghcr.io)"

# 推送 tag
git push origin v1.0.1

# 强制更新已有 tag（谨慎使用）
git tag -a v1.0.1 -f -m "GoalOS v1.0.1 — 更新说明"
git push origin v1.0.1 -f
```

### 3.2 Tag 命名

- 格式：`v<MAJOR>.<MINOR>.<PATCH>`
- 示例：`v1.0.0`、`v1.0.1`、`v1.1.0`、`v2.0.0`
- 禁止：`v1`、`release-1`、`1.0.1`

### 3.3 Tag 触发

推送 tag 时自动触发 GitHub Actions：
- 运行全量测试（`go test ./...`）
- 构建 Docker 镜像并发布到 ghcr.io
- 构建多平台二进制文件

## 4. Release 管理

### 4.1 创建 Release

```bash
# 方式 1：GitHub Web UI
# 打开 https://github.com/dotnet010/GoalOS/releases/new
# 选择 tag → 填写 Release Notes → 上传二进制 → 发布

# 方式 2：GitHub CLI
gh release create v1.0.1 \
  --title "GoalOS v1.0.1" \
  --notes-file RELEASE.md \
  goalos-darwin-amd64.tar.gz \
  goalos-darwin-arm64.tar.gz \
  goalos-linux-amd64.tar.gz \
  goalos-linux-arm64.tar.gz \
  goalos-windows-amd64.zip
```

### 4.2 Release 内容

每个 Release 必须包含：

| 内容 | 说明 |
|------|------|
| **Release Notes** | 本版本的新功能、修复、已知问题 |
| **多平台二进制** | darwin-amd64、darwin-arm64、linux-amd64、linux-arm64、windows-amd64 |
| **校验和** | SHA256 checksums（可选） |
| **Docker 镜像引用** | `ghcr.io/dotnet010/goalos:v1.0.1` |

### 4.3 多平台构建

```bash
# 编译所有平台
for target in "darwin/amd64" "darwin/arm64" "linux/amd64" "linux/arm64" "windows/amd64"; do
  os=$(echo $target | cut -d/ -f1)
  arch=$(echo $target | cut -d/ -f2)
  ext=""; [ "$os" = "windows" ] && ext=".exe"
  GOOS=$os GOARCH=$arch go build -o goalos-daemon$ext ./cmd/goalos/
  GOOS=$os GOARCH=$arch go build -o goalos$ext ./cmd/goalos-cli/
  # 打包：macOS/Linux → tar.gz。Windows → zip
done
```

## 5. Packages (ghcr.io) 管理

### 5.1 镜像命名

```
ghcr.io/dotnet010/goalos:<tag>
```

示例：
- `ghcr.io/dotnet010/goalos:v1.0.1`
- `ghcr.io/dotnet010/goalos:latest`（最新稳定版）

### 5.2 Docker 构建

```dockerfile
# 多阶段构建
FROM golang:1.23-alpine AS builder
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o goalos-daemon ./cmd/goalos/

FROM alpine:3.21
COPY --from=builder /build/goalos-daemon /usr/local/bin/
HEALTHCHECK --interval=10s CMD curl -f http://localhost:18920/api/health
EXPOSE 18920
```

### 5.3 GitHub Actions 自动发布

推送 tag 时自动触发 `.github/workflows/docker-publish.yml`：

```yaml
on:
  push:
    tags: ["v*"]
```

流程：Checkout → Test → Docker Build → Push to ghcr.io

### 5.4 Packages 可见性

在仓库 Settings → Packages 中设置：
- **Public**：所有人可拉取
- **Private**：仅仓库协作者可拉取

## 6. 发布检查清单

每次发布前逐条确认：

| # | 检查项 | 命令 |
|---|--------|------|
| 1 | 全量测试通过 | `go test -count=1 -timeout 60s ./...` |
| 2 | 编译通过 | `go build ./...` |
| 3 | 开发日志已更新 | 检查里程碑、模块、测试数 |
| 4 | 验收规范已通过 | 行为验收清单逐条验证 |
| 5 | 版本号已更新 | 开发日志、用户手册 |
| 6 | 多平台编译通过 | 5 个平台全部 BUILD OK |
| 7 | Tag 已创建 | `git tag -l 'v*'` |
| 8 | Docker workflow 通过 | 检查 GitHub Actions |
| 9 | Packages 页面可见 | 检查 ghcr.io 页面 |
| 10 | Release Notes 完整 | 新功能、修复、已知问题 |
