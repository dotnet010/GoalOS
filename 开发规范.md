# GoalOS 开发规范

> **基准规范**：[Google Go Best Practices](https://google.github.io/styleguide/go/best-practices)  
> **补充规范**：GoalOS 架构原则（9 条产品原则 + 5 条设计执行规则）  
> **最后更新**：2026-06-20

---

## 1. 代码规范

### 1.1 格式化

- 所有 Go 代码必须通过 `gofmt` 格式化
- 使用 `goimports` 自动管理 import

### 1.2 命名

- 包名：小写。单数。简短。如 `eventbus`、`scheduler`
- 导出符号：驼峰。`EventBus`、`Publish`
- 未导出符号：小写驼峰。`handler`、`lastSeq`
- 接口：单方法接口加 `-er` 后缀。`Subscriber`、`Publisher`

### 1.3 错误处理

- 永远不忽略 error。`if err != nil { return err }`
- 错误信息小写开头。不以标点结尾
- 使用 `fmt.Errorf("...: %w", err)` 包装错误

### 1.4 并发

- 接受 `context.Context` 作为第一个参数
- goroutine 生命周期明确（谁启动谁负责关闭）
- 使用 channel 通信，不共享内存
- sync.Mutex 作用域最小化

## 2. 测试规范（GoalOS 强制）

### 2.1 测试优先（TDD）

**测试必须走在开发前面。** 先写测试用例，确认测试失败，再实现功能，确认测试通过。

### 2.2 测试层级（强制）

每个模块必须包含两个层级的测试：

| 层级 | 说明 | 验证什么 | 示例 |
|------|------|---------|------|
| **单元测试** | 纯函数/接口级别 | 代码逻辑正确 | `TestTransition_DraftToPlanned` |
| **集成测试** | 运行中模块间交互 | 系统行为正确 | `TestE2EGoalLifecycle`（真实 daemon 中验证） |

**规则：仅有单元测试的模块 = 未完成。** 单元测试验证"代码对"。集成测试验证"系统对"。二者缺一不可。

### 2.3 集成测试最低要求

每个核心模块必须具备以下集成测试：

| 模块 | 最低集成测试 |
|------|------------|
| EventBus | 发布事件→多个 handler 收到→EventStore 持久化 |
| StateStore | Append→重启→Replay→状态恢复 |
| Scheduler | GoalCreated→PlanRequested→MissionGenerated→ActionScheduled 完整链 |
| Governance | ActionScheduled→五引擎→ActionApproved（真实 Policy/Risk 判定） |
| MissionEngine | PlanRequested→Agent.Plan()→MissionGenerated→events.jsonl |
| PluginRunner | ActionApproved→子进程启动→ActionCompleted（非 stub） |
| GoalAgent | 真实 LLM API 调用→JSON 解析→MissionGraph（非 mock） |

### 2.4 Mock 使用限制

- **mock 仅用于外部依赖**（LLM API、网络、文件系统）。不用于内部模块
- 内部模块间测试**必须使用真实 EventBus/StateStore/Scheduler/Governance**
- 如果测试无法在真实模块上运行——说明模块耦合太高，需要重构

### 2.5 测试文件与覆盖率

- 测试文件与源文件同目录。`eventbus_test.go`
- 单元测试覆盖率 > 80%。集成测试另外计算
- CI 中强制检查

### 2.6 测试用例质量标准（R238）

每个测试必须能回答三个问题：

1. **验证了什么行为？** — 测试函数注释中必须说明"验证：用户做X→系统做Y→期望结果Z"
2. **能检测实现错误吗？** — 如果被测代码逻辑错了，这个测试会失败吗？如果不会→这不是测试
3. **覆盖了错误路径吗？** — 每个模块至少 1 个测试覆盖异常场景（失败/超时/资源耗尽）

### 2.7 按模块类型的最低测试要求（R239）

| 模块类型 | 最低单元测试 | 最低集成测试 | 必须覆盖的错误路径 |
|---------|------------|------------|-----------------|
| 核心 (EventBus/StateStore/Scheduler) | 5+ | 2+ | 崩溃恢复/并发/资源耗尽 |
| 决策 (Governance/MissionEngine) | 3+ | 1+ | 拒绝/超时/重试耗尽 |
| 执行 (PluginRunner/Executor) | 2+ | 1+ | 子进程崩溃/超时/IPC截断 |
| API (daemon/client) | 3+ | 1+ | 无效请求/daemon不可用 |
| 入口 (cmd/*) | — | 1+ | daemon启动失败/端口冲突 |

## 3. 项目结构

```
GoalOS/
├── cmd/goalos/           # 入口。main.go
├── internal/             # 不对外暴露的内部包
│   ├── eventbus/         # Event Bus
│   ├── scheduler/        # Scheduler（含 Recovery/Verifier/Runtime）
│   ├── statestore/       # State Store
│   ├── governance/       # Governance Layer（五引擎）
│   ├── missionengine/    # Mission Engine（含 Agent）
│   ├── contextengine/    # Context Engine
│   └── pluginrunner/     # Plugin Runner（IPC）
├── pkg/events/           # Event 类型定义。对外暴露
├── test/                 # 集成测试
├── go.mod
├── go.sum
├── 开发规范.md           # 本文档
└── README.md
```

## 4. 架构纪律

### 4.1 模块边界

- 跨 `internal/` 包调用必须通过 Event Bus
- 同包内函数调用不受限
- 禁止 `internal/scheduler` import `internal/governance`

### 4.2 Event Bus

- `Publish` 同步阻塞。handler 纯内存操作（微秒级）
- 重 I/O 由独立 goroutine 异步完成
- handler panic 不影响其他 handler

### 4.3 State Store

- events.jsonl 是 Source of Truth
- state.json 是可重建缓存
- 启动时回放增量事件

## 5. 验收

所有里程碑验收必须遵循 [验收规范](验收规范.md)。验收由产品设计团队执行。

**验收不是检查"代码是否存在"——是验证"系统跑起来之后的行为是否符合设计"。**

## 6. Stub 与诚实标记（强制）

### 6.1 Stub 必须显式标注

任何临时实现（stub/hardcode/placeholder）必须在代码中标注：
```go
// W3: 待完成 — 当前为临时占位。完成期限：W3。
```

### 6.2 禁止虚假完成

**代码中仍有 W*/TODO/stub/不验证 标记的模块，开发日志不得标记为 ✅ 已完成。** Stub 是未完成的诚实信号——不是"差不多完成"。

### 6.3 开发者不得自审

写代码的人不能给自己的模块在开发日志中标记 ✅。状态更新必须经另一人对照 Stub 审计（验收规范 §7）验证。

## 7. Code Review

每个模块完成后，必须经过至少一人 Review：

- 架构纪律检查（模块边界、Event Bus 通信）
- 测试覆盖率检查
- 错误处理完整性
- 并发安全

---

## 7. 注释规范（强制）

### 6.1 语言

- **所有注释必须使用中文。** 包注释、函数注释、关键逻辑注释——一律中文。
- 注释与代码同等重要。缺少注释的代码视为未完成。

### 6.2 覆盖率

- 每个包必须有包注释（`// Package xxx 实现...`）
- 每个导出函数必须有函数注释（`// FuncName 做什么`）
- 关键算法（决策树、状态转换、风险评分）必须有行内注释解释逻辑
- 目标：注释覆盖率 > 10%

### 6.3 格式

- 包注释：`// Package xxx 实现 GoalOS 的...`
- 函数注释：`// FuncName 功能描述。`
- 设计依据注释：`// 设计依据：05 架构文档 §X。`

## 8. 开发日志维护（强制）

### 7.1 更新时机

- 每完成一个模块→更新开发日志.md
- 每完成一个 API 端点→更新接口清单
- 每完成一个里程碑→更新里程碑状态
- 每通过一个测试→更新测试统计

### 7.2 格式

- 使用 Markdown 表格
- 状态使用 emoji：✅ 完成 / 🟢 通过 / ⬜ 待开始 / ❌ 失败
