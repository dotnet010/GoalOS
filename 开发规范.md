# 标准开发规范

> **版本**: v2.0  
> **适用范围**: 所有软件项目。核心原则项目无关，Go 语言规则为 GoalOS 具体实现。  
> **哲学基础**: 基于第一性原理的开发纪律——不是"怎么写代码"，是"怎么确保写出来的代码是对的"。  
> **最后更新**: 2026-06-22

---

## 1. 开发哲学

### 1.1 三层正确性

```
代码正确 (Code Correct)   → 编译通过 + 单元测试通过
行为正确 (Behavior Correct) → 集成测试通过 + 端到端验证通过
设计正确 (Design Correct)   → 架构对齐 + 第一性原理不违背
```

**规则**: 三层任一不满足 → 模块标记 ❌。三层全部满足 → 标记 ✅。

### 1.2 第一性原理检查清单

每个模块、每个 PR、每个里程碑必须能回答：

| 原则 | 检查问题 | 不合格信号 |
|------|---------|-----------|
| 目标中心 | 用户需要知道这个模块的存在吗？ | API 响应暴露内部 action_type/capability 名称 |
| 权限隔离 | 这个动作经过 Governance 了吗？ | 日志中 ActionCompleted 前无对应的 ActionApproved |
| OTA 循环 | 系统是先观察再思考再执行的吗？ | Observer 未采集环境状态就进入 Plan |
| 极简核心 | 这个代码放在 Kernel 里还是 Plugin 里？ | Kernel 模块 import 了 LLM/Agent/Shell/Browser |

## 2. 开发流程

### 2.1 完整开发循环

```
  设计 (Architect + PM)
    ↓ R-xxx 决议编号
  测试 (Engineer)
    ↓ 测试先于实现
  实现 (Engineer)
    ↓ 
  自查 (Engineer)           ← 编译/测试/自查清单
    ↓
  Review (另一 Engineer)     ← 架构纪律/测试覆盖/并发安全
    ↓
  验收 (PM + Architect)      ← 行为验收 + 接线检查
    ↓
  点火 (Jobs + All)          ← 真实环境端到端
    ↓
  ✅ 完成
```

### 2.2 阶段闸口

| 阶段 | 准入条件 | 准出条件 |
|------|---------|---------|
| 设计 | 需求文档 | R-xxx 决议 + 架构评审通过 |
| 实现 | 测试用例已写 | 编译通过 + 测试通过 + 自查清单 ✅ |
| Review | 自查通过 | Review 通过 + Stub 审计清零 |
| 验收 | Review 通过 | 行为验收清单全部通过 + 接线检查通过 |
| 发布 | 验收通过 | 点火测试通过 + 四人签字 |

### 2.3 禁止事项

- ❌ 开发者自审自批（Review 必须另一人执行）
- ❌ 跳过任何阶段的闸口
- ❌ 修复一个 bug 后只重跑失败的测试（必须重跑全部）
- ❌ 测试用 mock 替代内部模块（仅外部依赖可用 mock）

## 3. 代码规范

### 3.1 命名

| 规则 | 示例 |
|------|------|
| 包名：小写、单数、简短 | `eventbus`, `scheduler` |
| 导出符号：驼峰 | `EventBus`, `Publish` |
| 未导出：小写驼峰 | `handleGoalCreated` |
| 接口：单方法加 `-er` | `Subscriber`, `Publisher` |

### 3.2 错误处理

- 永远不忽略 error
- 错误信息小写开头，不以标点结尾
- 使用 `fmt.Errorf("context: %w", err)` 包装
- 每个模块 ≥ 1 个 error path 测试

### 3.3 并发（GoalOS 强制）

| 规则 | 来源 |
|------|------|
| `sync.Mutex` 保护所有共享 map | C顾问发现 Scheduler 4 个 map DATA RACE |
| 锁在 `publish` 前释放（避免 EventBus 同步分发死锁） | 同一轮修复 |
| 全局计数器使用 `sync/atomic` 或 `atomic.Int64` | HTTP handler 并发访问 goalCounter |
| `-race` 必须加入 CI | 验收审计发现 CI 缺 race 检测 |

### 3.4 格式化

- Go: `gofmt` + `goimports`
- CI: `go vet ./...` 必须通过

## 4. 测试规范

### 4.1 测试层级（四层，缺一不可）

| 层级 | 验证什么 | 最低要求 | 失败后果 |
|------|---------|---------|---------|
| **单元测试** | 函数/接口逻辑 | 每模块 ≥ 1 error path | 模块 ❌ |
| **集成测试** | 模块间真实交互（非 mock） | 每跨模块调用链 ≥ 1 | 模块 ❌ |
| **契约测试** | 跨模块数据格式兼容 | 每格式依赖 ≥ 1 往返测试 | 模块 🟡 |
| **接线测试** | main/启动流程完整 | ≥ 1 启动序列测试 | 启动 ❌ |

### 4.2 测试质量标准

| # | 标准 | 验证方法 |
|---|------|---------|
| 1 | 每个测试能一句话回答"验证了什么行为" | 抽查：随机 3 个测试→问开发 |
| 2 | 每个模块 ≥ 1 个非 happy-path 测试 | 检查 error path 分支覆盖 |
| 3 | 内部模块间必须真实调用（非 mock） | EventBus/StateStore/Scheduler 实例必须真实 |
| 4 | `-race` 检测零警告 | CI 强制 |

### 4.3 Mock 限制

- ✅ 可以 mock: 外部网络 API、文件系统、OS 命令
- ❌ 不可以 mock: EventBus、StateStore、Scheduler、Governance、PluginRunner
- **理由**: 如果测试无法在真实模块上运行 → 模块耦合太高 → 需要重构

### 4.4 测试文件组织

- 测试文件与源文件同目录: `eventbus_test.go`
- 集成测试另放 `test/` 目录
- 测试函数名: `Test<模块>_<场景>_<预期>` 如 `TestGovernance_PolicyDeniesAction`

## 5. 架构纪律

### 5.1 模块边界

```
✅ Mission Engine → EventBus.Publish(PlanRequested) → Scheduler 订阅
❌ Mission Engine → import "internal/scheduler" → 直接调用
```

- 跨 `internal/` 包: **必须通过 Event Bus**
- 同包内: 直接函数调用
- 插件调用: Plugin 接口契约 (`Agent.Plan()`, `Channel.Send()`)

### 5.2 事件规范

- `Publish` 同步阻塞，handler 纯内存操作（微秒级）
- 重 I/O 由独立 goroutine 异步完成
- handler panic 不影响其他 handler
- 每个事件类型在事件拓扑表中有唯一 Publisher

### 5.3 State Store

- `events.jsonl` = Source of Truth（append-only, fsync）
- `state.json` = 可重建缓存（原子写入: tmp + rename）
- 启动时: 加载最新 snapshot + 回放增量 events → O(1)

## 6. 质量门禁

### 6.1 代码提交前（Engineer 自查）

```bash
grep -rn "不验证" internal/ cmd/ --include="*.go" | grep -v "_test.go"  # 必须返回空
go vet ./...          # 必须通过
go test -race ./...   # 必须通过
```

### 6.2 PR Review 前（另一 Engineer）

- [ ] Stub 审计: `不验证` 标记 = 0
- [ ] 死代码检查: 每个 internal/ 模块在 main.go 有调用
- [ ] 跨模块契约: 生产者-消费者格式一致
- [ ] 并发安全: `-race` 通过

### 6.3 里程碑验收前（PM + Architect）

- [ ] 行为验收清单全部通过
- [ ] 接线检查: 模块接入/事件对齐/配置-行为对齐
- [ ] 点火测试: daemon 真实启动 → curl → 检查日志

### 6.4 发布前（Jobs + All）

- [ ] 四人全部签字
- [ ] 开发日志 Stub 计数器已更新
- [ ] Release Notes 已写

## 7. 反模式清单（来自真实事故）

| # | 反模式 | 事故 | 教训 |
|---|--------|------|------|
| 1 | `cmd.Env = []string{...}` 替换而非追加 | macOS `fork/exec: no such file or directory` | 子进程环境变量必须 `append(os.Environ(), ...)` |
| 2 | stdout 读取 goroutine 在 write 之后启动 | 插件输出丢失 (0 bytes captured) | 先启动 reader，再 write execute 消息 |
| 3 | 跨模块 token 格式不兼容 | issueToken(HMAC-SHA256) vs VerifyToken(HS256) | 契约测试: 同一模块签发+验证的往返测试 |
| 4 | `RegisterCapabilities` 只在测试中调用 | 生产环境所有 Action 被拒绝 | 接线检查: 每个 Register/Start 在 main.go 有调用 |
| 5 | `_ = ctxEng` 丢弃实例 | ContextEngine 代码完整但从未启动 | 死代码检测: grep 每个模块的 Start/Register 调用 |
| 6 | builtin 能力集永久无条件放行 | Capability Engine 形同虚设 | 标注消除期限: `// W3: 删除 builtin, 改为真实 Plugin 声明` |
| 7 | 四个 map 无 `sync.Mutex` | DATA RACE → `fatal: concurrent map writes` | 所有共享 map 必须加锁，锁在 publish 前释放 |
| 8 | List 端点不 merge actionResults | 用户轮询 API 看到空结果 | 读路径和写路径对齐: TrackResult 存了 → List 要能读出来 |
| 9 | BPF jeq 跳转偏移硬编码 Jt=1 | seccomp 全部 KILL——上线即炸 daemon | 动态计算: `Jt = N - i`，每次新增 syscall 不改变 |
| 10 | 架构检查 Jt/Jf 反转 | 正确架构被 KILL，错误架构放行 | 仿真验证：BPF/状态机用独立脚本周遊所有分支 |
| 11 | 安全闸口用"空=跳过"语义 | 空签名 → 跳过验签 → Plugin 绕过安全加载 | 安全闸口统一用 fail-closed: 缺失/空/异常 → 拒绝 |
| 12 | stub回退绕过全部安全控制 | 执行失败→stub→不经过Policy/Capability/Risk | 执行失败=ActionFailed。删除stub回退路径 |
| 13 | Token scope/goal_id/action_id绑定未校验 | Token授权fs.read的Plugin可执行shell.execute | Token验证后检查Capabilities覆盖actionType |
| 14 | `_, ok := map[key]`丢弃值→下游收到空字段 | HandleApprove丢弃PendingApproval→goal_id=""→Goal永不完成 | map查值时必须捕获value |
| 15 | 关键词匹配≠目标理解 | "魔方"→预写模板→六面彩色盒子(非魔方) | Think阶段必须推理语义 |
| 16 | Agent硬编码领域知识模板 | 魔方HTML/水族箱HTML直接写在Agent代码中 | Agent只做Plugin路由,内容由Plugin生成 |
| 17 | 路由规则硬编码在Agent中 | "搜索"→web.search等关键词写死在代码里 | 路由规则=配置(文件/DB), Agent=纯调度器 |

## 8. Stub 与诚实标记

### 8.1 规则

```bash
# CI 中的 stub 检测
grep -rn "W[0-9] stub\|不验证\|TODO\|FIXME\|待实现\|临时\|简化实现\|hardcoded" \
  internal/ cmd/ --include="*.go" | grep -v "_test.go"
```

| 标记 | 含义 | 状态 |
|------|------|------|
| `W<X>: 待完成` | 临时占位，有明确消除期限 | 🟡 进行中 |
| `不验证` | **安全机制被跳过** | ❌ 阻塞——不得进入下一阶段 |
| `TODO` / `FIXME` | 已知缺陷 | 🟡 进行中 |
| `hardcoded` | 不可配置的值 | 🟡（需有明确消除计划） |

### 8.2 诚实性规则

- 代码中仍有 W*/TODO/stub 的模块 → 开发日志不得标记 ✅
- 过期 stub（超过标注期限未消除）→ 自动降级为 ❌
- 开发日志状态更新必须经另一人对照 Stub 审计结果验证

## 9. 设计决策记录

### 9.1 格式

每个设计决策必须有决议编号:

```
R-{编号}: {决议标题}
  日期: YYYY-MM-DD
  状态: ✅ 生效 / 📋 计划中 / ❌ 已废弃
  依据: {架构文档引用 或 会议编号}
  内容: {决议内容}
```

### 9.2 何时创建决议

- 架构变更（模块边界移动、接口修改）
- 技术选型（API provider、算法选择）
- 安全策略（capability 授权规则、风险阈值）
- **功能级决策也需要决议编号**（如搜索 API provider 从 Bing→Serper 的变更）

## 10. 文档维护

| 文档 | 维护者 | 更新时机 |
|------|--------|---------|
| **开发日志** | Engineer | 每完成一个模块/API/里程碑 |
| **开发规范** | Architect | 每次发现新的反模式或流程问题 |
| **验收规范** | PM | 每次验收审计后发现缺失的检查项 |
| **发布规范** | PM | 每次发布流程变更 |
| **架构文档** | Architect | 每次架构变更（伴随 R-xxx 决议） |

**规则**: 代码变更 → 对应文档同 PR 更新。不接受"代码先合，文档后补"。
