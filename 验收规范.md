# 标准验收流程规范

> **版本**: v2.0  
> **适用范围**: 所有软件项目。项目无关——可复用于任何语言、任何架构。  
> **哲学基础**: 验收不是检查"代码是否存在"。验收是验证**系统跑起来之后的行为是否符合设计**。  
> **核心规则**: 任一验收项不通过 → 该里程碑标记为 ❌ 未完成。修复后重跑全部验收项。

---

## 1. 验收原则

1. **代码存在 ≠ 功能完成**。测试通过 ≠ 行为正确。编译成功 ≠ 系统可用。
2. **没有"自审通过"**。写代码的人不能给自己的模块标记 ✅。每一层都需要独立角色验收。
3. **点火验收不可跳过**。必须在真实运行环境中验证至少一个端到端场景。
4. **修复一个 bug → 重跑全部验收**。不是只重跑失败那一条——修复可能引入回归。
5. **文档是验收依据**。没有文档的功能 = 无法验收的功能。架构文档是验收的"契约"。

---

## 2. 角色与职责

| 角色 | 验收职责 | 签字域 |
|------|---------|--------|
| **产品负责人** (PM/Jobs) | 愿景对齐、用户体验、功能完整性、端到端行为 | 点火验收 + 最终签字 |
| **软件架构师** | 架构合规、设计执行规则、跨模块契约、技术债 | 架构对齐 + 接线检查 |
| **产品经理** | 验收标准覆盖、用户旅程完整、错误路径体验 | 行为验收清单 |
| **资深研发工程师** | 代码质量、测试覆盖、并发安全、性能基准 | 代码验收 + 测试验收 |

**禁止事项**:
- 开发者不得给自己的模块标记 ✅
- PM 不得跳过架构师的接线检查签字
- 架构师不得在产品负责人之前签最终验收

---

## 3. 验收四阶段

```
  Phase 1: 代码验收 (Engineer)
      ↓
  Phase 2: 测试验收 (Engineer + Architect)
      ↓
  Phase 3: 行为验收 (PM + Architect)
      ↓
  Phase 4: 点火验收 (Jobs + All)
      ↓
  ✅ 里程碑通过
```

每个阶段是一个闸口。前一阶段不通过 → 不得进入下一阶段。

---

## 4. Phase 1: 代码验收

**执行人**: 资深研发工程师  
**验收依据**: 开发规范、架构文档

### 4.1 编译检查

| # | 检查项 | 方法 | 通过标准 |
|---|--------|------|---------|
| 1 | 全量编译 | `{build_command}` | 零错误、零警告 |
| 2 | 静态分析 | `{lint_command}` | 零 error 级别问题 |
| 3 | 跨平台编译 | 目标平台列表逐一编译 | 全部通过 |

### 4.2 Stub 审计

```bash
# 通用 stub 检测模式（项目可自定义关键词）
grep -rn "stub\|TODO\|FIXME\|不验证\|待实现\|临时\|简化实现\|hardcoded" \
  {source_dir} --include="*.{ext}" | grep -v "_test.{ext}"
```

| 标记类型 | 含义 | 状态 |
|---------|------|------|
| `stub` / `待实现` | 临时占位，承诺将来完成 | 🟡 进行中 |
| `不验证` | 安全机制被跳过 | ❌ 阻塞——不得进入下一阶段 |
| `TODO` / `FIXME` | 已知缺陷 | 🟡 进行中 |
| `hardcoded` | 不可配置的值 | 🟡 进行中（需有明确消除计划） |

**规则**: 任何 `不验证` 标记 = 阻塞。全部 stub 清零后模块才能标记 ✅。

### 4.3 死代码检测

检查每个模块的入口函数是否被 `main` 或初始化流程调用。未被调用的 exported 函数/模块 → 标记为 🟡（"代码存在但未接入"）。

---

## 5. Phase 2: 测试验收

**执行人**: 资深研发工程师 + 软件架构师  
**验收依据**: 测试标准

### 5.1 测试层级

| 层级 | 定义 | 最低要求 | 工具 |
|------|------|---------|------|
| **单元测试** | 验证单个函数/接口逻辑 | 每个模块 ≥ 1 个 error path 测试 | `{test_runner}` |
| **集成测试** | 验证模块间真实交互（非 mock） | 每个跨模块调用链 ≥ 1 个 | `{test_runner}` |
| **契约测试** | 验证跨模块数据格式兼容性 | 每个跨模块格式依赖 ≥ 1 个 | 序列化-反序列化往返测试 |
| **接线测试** | 验证 main/启动流程完整 | ≥ 1 个启动序列测试 | 启动 → health check → 日志 |

### 5.2 测试质量

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 并发安全 | `{test_runner} -race` 零警告 |
| 2 | 错误路径 | 每个模块至少 1 个非 happy-path 测试 |
| 3 | Mock 限制 | 仅外部依赖（网络/数据库/OS）可用 mock。内部模块间必须真实调用 |
| 4 | 测试可解释 | 每个测试函数能一句话回答"验证了什么行为" |

### 5.3 测试不通过后果

- 任一条不通过 → 模块标记 ❌
- 集成测试缺失 → 单元测试 100% 覆盖也不通过
- 接线测试缺失 → 启动流程标记 ❌

---

## 6. Phase 3: 行为验收

**执行人**: 产品经理 + 软件架构师  
**验收依据**: 产品需求文档、架构文档

### 6.1 行为验收清单模板

| # | 验收场景 | 验证方法 | 通过标准 |
|---|---------|---------|---------|
| AC-1 | 系统启动 | 启动进程 → 检查健康端点 | 返回预期状态 |
| AC-2 | 核心功能端到端 | 触发核心用户旅程 | 预期输出 + 日志无异常 |
| AC-3 | 错误路径 | 触发失败场景 | 系统诚实失败（不伪造成功、不静默卡死） |
| AC-4 | 事件/审计链 | 检查事件日志 | 完整事件链（触发→处理→结果） |
| AC-5 | 安全闸口 | 尝试绕过安全检查 | 被正确拒绝 + 审计记录 |
| AC-6 | 超时反馈 | 等待超时阈值 | 系统在超时内给出用户可见的状态变化 |

### 6.2 架构对齐检查

| # | 检查项 | 方法 |
|---|--------|------|
| 1 | 模块边界 | 跨模块调用是否全部通过事件总线/API |
| 2 | 设计执行规则 | 决策模块有决策算法？写入路径有读取路径？ |
| 3 | 安全边界 | OS 级别强制执行（非代码约定）？ |
| 4 | 文档-代码一致性 | 核心结构体/接口与架构文档定义一致 |

---

## 7. Phase 4: 点火验收（Live Fire）

**执行人**: 产品负责人 (Jobs) + 全团队  
**验收依据**: 愿景与第一性原理

### 7.1 点火测试（强制。不可跳过。）

```
1. 编译生产二进制
2. 在生产等效环境启动系统
3. 发送真实用户请求（非 mock、非测试数据）
4. 检查: 日志、事件存储、API 响应、用户可见状态
5. 验证: 整个链路无 stub 介入、无静默失败
```

**通过标准**: 日志中无 `stub`/`fallback` 关键词，端到端延迟在 P95 阈值内，用户可见状态在预期时间内变化。

### 7.2 第一性原理验证

| 原则 | 验证问题 | 验收标准 |
|------|---------|---------|
| 以目标为中心 | 用户是否只需说"我要什么"？ | 输入为自然语言 Goal，系统不暴露内部 Action/Plugin 细节 |
| 权限隔离 | 所有执行是否经过策略控制？ | 每个 Action 在 events 中有关联的 Governance 决策事件 |
| 观察-思考-执行 | 系统是否按 O-T-A 循环运作？ | 日志中可观察到 Observer→Thinker→Executor 的事件序列 |
| 极简核心 | 内核是否只做调度和事件管理？ | 内核模块无 LLM/Agent/Shell/Browser 调用 |

## 8. 跨切面检查（每个 Phase 都需执行）

### 8.1 接线检查

| # | 检查项 | 方法 |
|---|--------|------|
| 1 | 模块接入 | 每个模块的入口函数在启动流程中有调用 |
| 2 | 跨模块契约 | 跨模块调用的输入输出格式一致（序列化-反序列化往返测试验证） |
| 3 | 事件生产者-消费者对齐 | 每个事件类型至少有一个非 Audit 的 subscriber |
| 4 | 配置-行为对齐 | 配置文件字段被代码读取和使用 |

### 8.2 回归清单

> **背景**: 验收审计发现"修好一层暴露下一层"是系统性模式（Capability 拒绝修好 → 暴露 Payload 转发缺失 → 暴露 stdout pipe 捕获失败）。

| # | 场景 | 检查方法 |
|---|------|---------|
| 1 | 新模块接入后 | 重跑完整点火测试——新模块可能改变事件流 |
| 2 | 安全闸口修改后 | 验证拒绝路径 + 放行路径两端 |
| 3 | 跨模块格式变更后 | 契约测试——producer/consumer 两端序列化兼容 |
| 4 | stub 消除后 | 验证下游模块能正确处理真实数据（而非 stub 假数据） |

## 9. 不通过处理

1. **记录**: 哪一条未通过、失败现象、日志片段、复现步骤
2. **分类**: 
   - 技术问题（代码 bug）→ 开发修复
   - 设计问题（架构缺陷）→ 团队讨论决议（产生 R-xxx 决议编号）
   - 流程问题（验收规范自身缺陷）→ 更新本文档
3. **修复后重跑全部**: 不是只重跑失败那一条。修复可能引入回归

---

## 10. 项目适配指南

本规范是项目无关模板。适配新项目时：

1. **替换 `{build_command}`** 为实际构建命令（如 `go build ./...`, `cargo build`, `npm run build`）
2. **替换 `{test_runner}`** 为实际测试命令（如 `go test -race`, `pytest`, `jest`）
3. **替换 `{source_dir}`** 为源码目录
4. **填充行为验收清单**（§6.1）为项目特定的用户旅程
5. **保持四阶段结构、角色职责、跨切面检查不变**——这些是流程骨架
