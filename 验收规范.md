# GoalOS 验收规范

> **版本**: v1.0  
> **最后更新**: 2026-06-20  
> **适用范围**: 所有里程碑验收。产品设计团队执行。

---

## 1. 验收原则

验收不是检查"代码是否存在"。验收是验证**系统跑起来之后的行为是否符合设计**。

**核心规则**：任一条行为验收项不通过 → 该里程碑标记为 ❌ 未完成。

---

## 2. 验收流程

```
1. 代码编译通过 (go build ./...)
2. 全部单元测试通过 (go test ./...)
3. 架构对齐检查（对照 05 架构文档）
4. 行为验收清单逐条验证（启动 daemon → 触发场景 → 检查日志/API 响应）
5. 全部通过 → ✅ 里程碑通过
   任一条不通过 → ❌ 打回开发
```

---

## 3. W1-W2 行为验收清单

| # | 验收场景 | 验证方法 | 通过标准 |
|---|---------|---------|---------|
| 1 | Daemon 启动 | `./goalos-daemon &` → `curl /api/health` | 返回 `{"status":"ok"}` |
| 2 | 创建 Goal | `curl -X POST /api/goals -d '{"goal":"test"}'` | 返回 201 + goal_id |
| 3 | events.jsonl 持久化 | 检查 `~/.goalos/events/<goal_id>/events.jsonl` | 文件存在。包含 GoalCreated 事件 |
| 4 | CLI 命令 | `./goalos health` | 输出 "GoalOS Daemon: 运行中" |

## 4. W3-W4 行为验收清单

| # | 验收场景 | 验证方法 | 通过标准 |
|---|---------|---------|---------|
| 1 | **GoalAgent 调用 LLM** | 创建 Goal → 检查 daemon 日志 | 日志包含 `[GoalAgent] planning goal:` |

| 5 | **Snapshot 冷启动恢复** | 创建 Goal → 重启 daemon → 检查状态 | Goal 状态正确恢复 |
| 6 | Plugin 发现 | daemon 日志 | 包含已发现 Plugin 列表 |

## 6. 测试验收（开发前执行）

### 6.1 测试编写标准

每个模块必须包含两层测试：
- **单元测试**：验证纯函数/接口逻辑。覆盖 > 80%
- **集成测试**：验证运行中模块间交互。使用真实 EventBus/StateStore/Scheduler（非 mock）

**仅有单元测试的模块 = 未完成。**

### 6.2 测试验收清单

| # | 验收项 | 验证方法 | 通过标准 |
|---|--------|---------|---------|
| 1 | 单元测试全部通过 | `go test ./...` | 0 失败 |
| 2 | 集成测试存在且通过 | 检查 `test/` 目录 | 每个核心模块有至少 1 个集成测试 |
| 3 | Mock 仅用于外部依赖 | 代码审查 | 内部模块间不使用 mock |
| 4 | 测试在开发之前编写 | 检查 git log | 测试 commit 在实现 commit 之前 |

### 6.3 测试不通过的后果

- 任一条不通过 → 该模块标记为 ❌ 未完成
- 集成测试缺失 → 即使单元测试 100% 覆盖也不通过

### 6.4 测试用例质量验收（R240）

| # | 验收项 | 通过标准 | 检查方法 |
|---|--------|---------|---------|
| 1 | 行为可解释 | 每个测试函数能回答"验证了什么行为" | 抽查：随机选 3 个测试→问开发"这个测试验证了什么" |
| 2 | 可检测错误 | 至少 1 个测试覆盖错误路径 | 检查每个模块是否有非 happy-path 测试 |
| 3 | 集成测试比例 | 集成测试数 ≥ 总测试数 × 0.2 | `grep -r "func Test" test/` 计数 |
| 4 | TDD 时序 | 测试 commit 在实现 commit 之前 | `git log --oneline` 检查时间顺序 |
| 5 | Mock 限制 | 内部模块间不使用 mock | 代码审查：EventBus/StateStore 实例必须真实 |

## 7. Stub 审计（强制。里程碑验收前必须执行）

### 7.1 规则

**任何包含以下标记的代码，对应的开发日志条目不得标记为 ✅ 已完成。**

```bash
grep -rn "W1 stub\|W[0-9]:\|W[0-9] \|不验证\|待实现\|TODO\|FIXME\|暂时\|临时\|简化实现\|hardcoded" \
  internal/ cmd/ --include="*.go" | grep -v "_test.go"
```

| 标记类型 | 含义 | 开发日志应标记为 |
|---------|------|---------------|
| `W1 stub` / `W3: 待实现` | 临时占位，承诺将来实现 | 🟡 进行中 |
| `不验证` | 安全机制被跳过 | ❌ 阻塞 |
| `TODO` / `FIXME` | 已知缺陷 | 🟡 进行中 |
| `hardcoded` | 硬编码，不可配置 | 🟡 进行中 |

**全部清除后，模块才能标记为 ✅。**

### 7.2 开发日志诚实性规则

| 规则 | 说明 |
|------|------|
| **禁止虚假 ✅** | 代码中仍有 W*/TODO/stub 标记的模块，开发日志不得点 ✅ |
| **Stub 必须标注期限** | 任何临时实现标注 `W<X>：待完成`。过期 stub 自动降级 ❌ |
| **独立验证** | 开发日志状态更新必须经另一人对照 Stub 审计结果验证 |
| **开发者不得自审** | 写代码的人不能给自己的模块标记 ✅ |

### 7.3 Stub 审计执行时机

- 每次里程碑验收前
- 每次开发日志更新前
- CI 流程中自动执行（`grep` 以非零退出码失败 = 有 Stub 残留）

## 8. 不通过的处理

1. 记录：哪一条未通过、失败现象、daemon 日志片段
2. 分类：技术问题（代码 bug）→ 开发修复。设计问题（架构缺陷）→ 团队讨论决议
3. 修复后重新执行**全部**行为验收清单——不是只重跑失败那一条
